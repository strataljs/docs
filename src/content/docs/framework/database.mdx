`@stratal/framework` provides a database module built on [ZenStack](https://zenstack.dev/) ORM. It supports multiple named connections, per-connection model slicing, and plugins for error handling, event emission, and schema switching.

## Setup

### Install dependencies

```bash
yarn add @stratal/framework
```

### Configure DatabaseModule

Use `DatabaseModule.forRoot()` to configure your database connections:

```typescript
import { Module } from 'stratal/module'
import { DatabaseModule } from '@stratal/framework/database'

@Module({
  imports: [
    DatabaseModule.forRoot({
      schema: mySchema,
      default: 'main',
      connections: [
        {
          name: 'main',
          dialect: () => new PostgresDialect({ pool }),
        },
      ],
    }),
  ],
})
export class AppModule {}
```

### Configuration options

| Option | Type | Description |
| --- | --- | --- |
| `schema` | `object` | Your ZenStack schema |
| `default` | `string` | Name of the default connection |
| `connections` | `ConnectionConfig[]` | Array of connection configurations |

Each connection accepts:

| Option | Type | Description |
| --- | --- | --- |
| `name` | `string` | Unique connection name |
| `dialect` | `() => Dialect` | Factory function returning a database dialect |
| `plugins` | `Plugin[]` | Optional array of database plugins |
| `slicing` | `{ includedModels: string[] }` | Optional model slicing configuration |

## Injecting the database

### Default connection

Inject the default database connection using `DI_TOKENS.Database`:

```typescript
import { Transient, inject, DI_TOKENS } from 'stratal/di'
import type { DatabaseService } from '@stratal/framework/database'

@Transient()
export class UsersService {
  constructor(
    @inject(DI_TOKENS.Database) private readonly db: DatabaseService,
  ) {}

  async findAll() {
    return this.db.user.findMany()
  }
}
```

### Named connections

Use `@InjectDB(name)` to inject a specific named connection:

```typescript
import { Transient } from 'stratal/di'
import { InjectDB, type DatabaseService } from '@stratal/framework/database'

@Transient()
export class AnalyticsService {
  constructor(
    @InjectDB('analytics') private readonly db: DatabaseService<'analytics'>,
  ) {}

  async trackEvent(event: string) {
    await this.db.analyticsEvent.create({ data: { event } })
  }
}
```

## Multi-connection support

Configure multiple connections with model slicing to control which models are available on each connection. Slicing configuration is auto-generated by the ZenStack plugin — you define which models belong to each connection using `@@connection("name")` in your `.zmodel` schema, and the plugin generates the `connectionSlicing` constant for you.

```typescript
import { connectionSlicing } from '../db/zenstack/slicing'

DatabaseModule.forRoot({
  schema: mySchema,
  default: 'main',
  connections: [
    {
      name: 'main',
      slicing: connectionSlicing.main,
      dialect: () => new PostgresDialect({ pool: mainPool }),
    },
    {
      name: 'analytics',
      slicing: connectionSlicing.analytics,
      dialect: () => new PostgresDialect({ pool: analyticsPool }),
    },
  ],
})
```

With slicing, `DatabaseService<'main'>` only exposes models assigned to the `main` connection, while `DatabaseService<'analytics'>` only exposes models assigned to `analytics`.

### How slicing works

1. Annotate models in your `.zmodel` schema with `@@connection("connectionName")`. You can optionally pass a second argument to specify a different database provider (e.g. `@@connection("analytics", "mysql")`):

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id    String @id @default(cuid())
  email String @unique
  posts Post[]

  @@connection("main")
}

model Post {
  id       String @id @default(cuid())
  title    String
  author   User   @relation(fields: [authorId], references: [id])
  authorId String

  @@connection("main")
}

model AnalyticsEvent {
  id        String   @id @default(cuid())
  event     String
  createdAt DateTime @default(now())

  @@connection("analytics")
}

model PageView {
  id   String @id @default(cuid())
  path String

  // Use a different provider for this connection
  @@connection("analytics", "mysql")
}
```

2. Add the plugin to your `.zmodel` schema:

```prisma
plugin stratal {
  provider = "@stratal/zenstack-plugin"
  output   = "./zenstack"   // output directory (relative to schema)
  default  = "main"         // default connection for models without @@connection
}
```

3. Run `zenstack generate` — the plugin auto-generates:
   - **Per-connection schema files** in `connections/{name}/schema.zmodel` — each schema includes only models for that connection, with a `datasource` block whose `url` is set to `env("{CONNECTION_NAME}_DATABASE_URL")` (e.g. `MAIN_DATABASE_URL`, `ANALYTICS_DATABASE_URL`). Make sure to define these environment variables.
   - **`slicing.ts`** — exports a `connectionSlicing` constant and the `StratalDatabase` module augmentation

4. Import `connectionSlicing` in your database config (no manual augmentation needed)

## Plugins

The `DatabaseModule` automatically registers the following plugins on every connection — no configuration needed:

- **ErrorHandlerPlugin** — Transforms ZenStack errors into Stratal `ApplicationError` instances with appropriate HTTP status codes.
- **EventEmitterPlugin** — Emits events before and after database operations, integrating with Stratal's [event system](/core-concepts/events/). See [Database Events](/framework/database-events/) for the full event pattern documentation.

### SchemaSwitcherPlugin (opt-in)

The only user-configurable plugin is `SchemaSwitcherPlugin`, which sets the PostgreSQL `search_path` for multi-tenant isolation:

```typescript
import { SchemaSwitcherPlugin } from '@stratal/framework/database'

{
  name: 'main',
  dialect: () => new PostgresDialect({ pool }),
  plugins: [new SchemaSwitcherPlugin()],
}
```

## Type-safe schema augmentation

When using the ZenStack plugin with `@@connection()` annotations, the `StratalDatabase` module augmentation is **auto-generated** in the `slicing.ts` file alongside the `connectionSlicing` constant. You do not need to write this manually.

The generated augmentation provides full type safety — `DatabaseService` and `DatabaseService<'analytics'>` return properly typed clients with only the models available on each connection.

## CLI commands

The `stratal-db` CLI manages database migrations and schema operations across connections.

### Migrations

```bash
# Run dev migrations
stratal-db migrate dev --connection main --name add_users_table

# Run dev migrations for all connections
stratal-db migrate dev --all-connections

# Deploy migrations to production
stratal-db migrate deploy --connection main

# Check migration status
stratal-db migrate status --connection main

# Reset database
stratal-db migrate reset --connection main
```

### Schema push

```bash
# Push schema to database (skips migration history)
stratal-db push --connection main

# Push all connections
stratal-db push --all-connections
```

### Common options

| Option | Alias | Description |
| --- | --- | --- |
| `--connection <name>` | `-c` | Target a specific connection |
| `--all-connections` | | Run against all configured connections |
| `--schema <path>` | | Path to the schema file (default: `schema.zmodel`) |
| `--name <name>` | `-n` | Migration name (for `migrate dev`) |
| `--cleanup` | | Remove generated per-connection schemas after execution (default: `true`) |
| `--no-cleanup` | | Keep generated per-connection schemas |
